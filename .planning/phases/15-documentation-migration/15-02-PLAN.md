## Header
- Title: "Migrating from v0.2.x to v0.3.0"
- Brief overview: v0.3.0 changes constructors from positional strings to options objects, adds WebSocket configuration exposure
- Link to CHANGELOG.md for full change list
- Note about backward compatibility: Python and Node.js string constructors still work but emit deprecation warnings, will be removed in v0.4.0

## Breaking Changes Summary
Table summarizing what changed:
| Change | Languages | Impact |
|--------|-----------|--------|
| Constructor API | All | Options object/kwargs instead of string |
| Health check default | All | `false` (was `true`) |
| Auth validation | All | Exactly one auth method required |

## Language-Specific Migration Guides

### Python
Before (v0.2.x):
```python
from marketdata_py import RestClient
client = RestClient("your-api-key")
client2 = RestClient.with_bearer_token("your-token")
ws = WebSocketClient("your-api-key")
```

After (v0.3.0):
```python
from marketdata_py import RestClient, WebSocketClient, ReconnectConfig, HealthCheckConfig
client = RestClient(api_key="your-api-key")
client2 = RestClient(bearer_token="your-token")
ws = WebSocketClient(
    api_key="your-api-key",
    reconnect=ReconnectConfig(max_attempts=10),
    health_check=HealthCheckConfig(enabled=True),
)
```

Migration steps:
1. Replace `RestClient("key")` with `RestClient(api_key="key")`
2. Replace `RestClient.with_bearer_token("token")` with `RestClient(bearer_token="token")`
3. Replace `RestClient.with_sdk_token("token")` with `RestClient(sdk_token="token")`
4. Replace `WebSocketClient("key")` with `WebSocketClient(api_key="key")`
5. (Optional) Add reconnect/health_check config if needed

Automated migration: Reference `migration/migrate-python.py` script

### Node.js / TypeScript
Before (v0.2.x):
```javascript
const { RestClient, WebSocketClient } = require('@fubon/marketdata-js');
const client = new RestClient('your-api-key');
const ws = new WebSocketClient('your-api-key');
```

After (v0.3.0):
```typescript
import { RestClient, WebSocketClient } from '@fubon/marketdata-js';
const client = new RestClient({ apiKey: 'your-api-key' });
const ws = new WebSocketClient({
  apiKey: 'your-api-key',
  reconnect: { maxAttempts: 10, initialDelayMs: 2000 },
  healthCheck: { enabled: true, intervalMs: 15000 },
});
```

Migration steps:
1. Replace `new RestClient('key')` with `new RestClient({ apiKey: 'key' })`
2. Replace `new WebSocketClient('key')` with `new WebSocketClient({ apiKey: 'key' })`
3. (Optional) Add reconnect/healthCheck config

Automated migration: Reference `migration/migrate-javascript.js` script

### Java
Before (v0.2.x):
```java
RestClient client = new RestClient("your-api-key");
```

After (v0.3.0):
```java
import tw.com.fugle.marketdata.*;
FugleRestClient client = FugleRestClient.builder()
    .apiKey("your-api-key")
    .reconnectOptions(new ReconnectOptions.Builder()
        .maxAttempts(10)
        .build())
    .build();
```

### Go
Before (v0.2.x):
```go
client, _ := marketdata.NewRestClientWithApiKey("your-api-key")
```

After (v0.3.0):
```go
client, err := marketdata.NewFugleRestClient(
    marketdata.WithApiKey("your-api-key"),
    marketdata.WithReconnectOptions(marketdata.ReconnectOptions{
        MaxAttempts: 10,
    }),
)
```

### C#
Before (v0.2.x):
```csharp
var client = new RestClient("your-api-key");
```

After (v0.3.0):
```csharp
var client = new RestClient(new RestClientOptions {
    ApiKey = "your-api-key"
});
// Or still supported: new RestClient("your-api-key")
```

## Common Issues
- "ValueError: Provide exactly one of..." - You provided zero or multiple auth methods
- "ConfigError: max_attempts must be >= 1" - Config validation now happens at construction
- Health check not running - Default changed to false, explicitly enable if needed

### ReconnectConfig / ReconnectOptions

Reference table (source: core/src/websocket/reconnection.rs constants):
| Option | Type | Default | Min | Max | Description |
|--------|------|---------|-----|-----|-------------|
| max_attempts | u32/int/number | 5 | 1 | - | Maximum reconnection attempts |
| initial_delay_ms | u64/int/number | 1000 | 100 | - | Initial backoff delay (ms) |
| max_delay_ms | u64/int/number | 60000 | >= initial_delay_ms | - | Maximum backoff cap (ms) |

Language-specific examples for each:

Python:
```python
from marketdata_py import ReconnectConfig
config = ReconnectConfig(max_attempts=10, initial_delay_ms=2000, max_delay_ms=120000)
```

JavaScript/TypeScript:
```typescript
const reconnect: ReconnectOptions = {
  maxAttempts: 10,
  initialDelayMs: 2000,
  maxDelayMs: 120000,
};
```

Java:
```java
ReconnectOptions reconnect = new ReconnectOptions.Builder()
    .maxAttempts(10)
    .initialDelayMs(2000L)
    .maxDelayMs(120000L)
    .build();
```

Go:
```go
marketdata.WithReconnectOptions(marketdata.ReconnectOptions{
    MaxAttempts:    10,
    InitialDelayMs: 2000,
    MaxDelayMs:     120000,
})
```

C#:
```csharp
var reconnect = new ReconnectOptions {
    MaxAttempts = 10,
    InitialDelayMs = 2000,
    MaxDelayMs = 120000,
};
```

### HealthCheckConfig / HealthCheckOptions

Reference table (source: core/src/websocket/health_check.rs constants):
| Option | Type | Default | Min | Max | Description |
|--------|------|---------|-----|-----|-------------|
| enabled | bool | false | - | - | Enable WebSocket health monitoring |
| interval_ms | u64/int/number | 30000 | 5000 | - | Ping interval (ms) |
| max_missed_pongs | u64/int/number | 2 | 1 | - | Missed pongs before reconnect |

Additional constraint: timeout (implicit) must be less than interval.

Language-specific examples for each (same format as ReconnectConfig).

### Authentication Options

Reference table:
| Option | Type | Description |
|--------|------|-------------|
| api_key / apiKey / ApiKey | string | Fugle API key |
| bearer_token / bearerToken / BearerToken | string | Bearer token |
| sdk_token / sdkToken / SdkToken | string | SDK token |
| base_url / baseUrl / BaseUrl | string (optional) | Custom API base URL |

Constraint: Exactly one of api_key, bearer_token, sdk_token must be provided.

### Validation Error Messages
Document the error messages users will see for common validation failures:

- "initial_delay_ms must be >= 100" (too-small delay)
- "max_delay_ms must be >= initial_delay_ms" (illogical range)
- "interval_ms must be >= 5000" (too-frequent health checks)
