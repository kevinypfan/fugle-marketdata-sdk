Create config structs in package `marketdata_uniffi`:

```go
// ReconnectConfig configures WebSocket reconnection behavior.
// Zero values for fields mean "use default".
type ReconnectConfig struct {
    // MaxAttempts is the maximum number of reconnection attempts (default: 5, min: 1)
    MaxAttempts uint32
    // InitialDelayMs is the initial reconnection delay in milliseconds (default: 1000, min: 100)
    InitialDelayMs uint64
    // MaxDelayMs is the maximum reconnection delay in milliseconds (default: 60000)
    MaxDelayMs uint64
}

// HealthCheckConfig configures WebSocket health check behavior.
// Zero values for fields mean "use default".
type HealthCheckConfig struct {
    // Enabled controls whether health check is active (default: false)
    Enabled bool
    // IntervalMs is the interval between ping messages in milliseconds (default: 30000, min: 5000)
    IntervalMs uint64
    // MaxMissedPongs is the maximum missed pongs before disconnect (default: 2, min: 1)
    MaxMissedPongs uint64
}
```

Document default values in comments. Note that Go's zero-value for bool is false which matches DEFAULT_HEALTH_CHECK_ENABLED.

**options.go:**
Create functional option types and With* functions:

Define two option types — one for REST, one for WebSocket:
```go
// RestOption configures a RestClient
type RestOption func(*restConfig) error

// WebSocketOption configures a WebSocketClient
type WebSocketOption func(*wsConfig) error
```

Internal config structs (unexported):
```go
type restConfig struct {
    apiKey      string
    bearerToken string
    sdkToken    string
    baseUrl     string
}

type wsConfig struct {
    apiKey      string
    bearerToken string
    sdkToken    string
    baseUrl     string
    endpoint    WebSocketEndpoint
    reconnect   *ReconnectConfig
    healthCheck *HealthCheckConfig
}
```

Create With* functions that work for BOTH RestOption and WebSocketOption. To avoid duplication, create common auth setters using a shared interface or create separate functions for each. Simplest approach: separate functions for REST and WS, but since they share the same fields, use generics or define twice.

Recommended approach: Define common auth config struct and embed it:
```go
type authConfig struct {
    apiKey      string
    bearerToken string
    sdkToken    string
    baseUrl     string
}
```

Then define functions that return both option types. Since Go doesn't have union types, provide separate With* functions that return RestOption and WsOption variants, or more idiomatically, create a single Option type that works with both (since both configs embed the same auth fields).

Actually, the cleanest approach: Use a SINGLE `Option` type with a combined config. Let `NewRestClient` and `NewWebSocketClient` both accept `...Option`:

```go
type Option func(*clientConfig) error

type clientConfig struct {
    apiKey      string
    bearerToken string
    sdkToken    string
    baseUrl     string
    endpoint    WebSocketEndpoint
    reconnect   *ReconnectConfig
    healthCheck *HealthCheckConfig
}
```

With* functions:
- `WithApiKey(key string) Option` — validates non-empty, sets apiKey
- `WithBearerToken(token string) Option` — validates non-empty, sets bearerToken
- `WithSdkToken(token string) Option` — validates non-empty, sets sdkToken
- `WithBaseUrl(url string) Option` — sets baseUrl
- `WithEndpoint(ep WebSocketEndpoint) Option` — sets endpoint (WebSocket only, defaults to Stock)
- `WithReconnect(cfg ReconnectConfig) Option` — stores reconnect config pointer
- `WithHealthCheck(cfg HealthCheckConfig) Option` — stores health check config pointer

Each With* function returns a closure that modifies the config. Non-empty validation for auth tokens (return error if empty string).

**client.go:**
Create idiomatic Go client constructor functions in package `marketdata_uniffi`:

**`NewFugleRestClient(opts ...Option) (*RestClient, error)`:**
1. Create default `clientConfig{}`
2. Apply all options (return error if any option returns error)
3. Validate exactly one auth:
   - Count non-empty auth fields
   - If 0: return `errors.New("provide exactly one of: WithApiKey, WithBearerToken, or WithSdkToken")`
   - If >1: return `errors.New("provide exactly one of: WithApiKey, WithBearerToken, or WithSdkToken")`
4. Call the appropriate UniFFI-generated constructor:
   - apiKey set: `NewRestClientWithApiKey(cfg.apiKey)` (this returns `*RestClient, error`)
   - bearerToken set: `NewRestClientWithBearerToken(cfg.bearerToken)`
   - sdkToken set: `NewRestClientWithSdkToken(cfg.sdkToken)`
5. If err from UniFFI, wrap and return: `fmt.Errorf("failed to create client: %w", err)`
6. baseUrl: Store but don't apply (same as Python/Node.js — UniFFI RestClient doesn't expose base_url setter). Add TODO comment.
7. Return client

**`NewFugleWebSocketClient(listener WebSocketListener, opts ...Option) (*StreamingClient, error)`:**
NOTE: Look at existing `NewStreamingClient` and `NewStreamingClientWithEndpoint` in channel_wrapper.go. The new function should:
1. Create default `clientConfig{endpoint: WebSocketEndpointStock}`
2. Apply all options
3. Validate exactly one auth (same as REST)
4. Create channel-based wrapper (same as existing NewStreamingClient pattern)
5. Call appropriate UniFFI constructor based on auth method and endpoint
   - NOTE: Current UniFFI WebSocketClient constructors only accept api_key string. For bearerToken/sdkToken, add TODO comment (same pattern as other languages)
6. Store reconnect and healthCheck configs for future use (not yet wired to core)
7. Return StreamingClient

Also provide a variant without listener for direct WebSocket access:
**`NewFugleWebSocketClientDirect(listener WebSocketListener, opts ...Option) (*WebSocketClient, error)`**
Actually, keep it simple — just provide `NewFugleWebSocketClient` that returns `*StreamingClient` (the channel-based wrapper). Users who want raw access can use the UniFFI-generated functions directly.
