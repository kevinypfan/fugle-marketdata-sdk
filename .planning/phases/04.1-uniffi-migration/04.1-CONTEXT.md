# Phase 4.1: UniFFI Migration - Context

**Gathered:** 2026-01-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate C# binding from csbindgen to UniFFI and add Go binding via uniffi-bindgen-go. Python binding (PyO3) and Node.js binding (napi-rs) remain unchanged — they're already mature and working. UniFFI provides a single UDL definition that generates both C# and Go bindings, reducing maintenance burden for these languages.

</domain>

<decisions>
## Implementation Decisions

### API Surface Design
- Expose **both sync and async** variants of REST methods in UDL
- Use **flat enum** for errors (Network, Auth, RateLimit, Parse, etc.) — not hierarchical
- WebSocket supports **both patterns**: callback interface AND channel/iterator — user chooses
- Response models as **full structs** with all fields exposed — direct property access

### Migration Strategy
- **Keep PyO3 for Python** — already mature, don't touch py/ crate
- **Replace csbindgen with UniFFI for C#** — delete cs/ crate entirely after migration
- **Include Go** in this phase — UniFFI makes it nearly free once UDL exists
- **Breaking changes OK** for C# — new UniFFI API can differ from current csbindgen wrapper
- **Delete cs/ crate and C# wrapper** after UniFFI C# binding is validated
- **Partial FubonNeo compatibility** for C# — match key patterns (method names, async style) but allow structural differences
- Use **thin C# wrapper** if needed to adapt UniFFI output toward FubonNeo-compatible interface

### Async Behavior
- **Modern typed pattern** for both C# and Go:
  - REST: `Task<Quote>` / `(*Quote, error)` — parsed models, not raw responses
  - WebSocket: typed callback interface with strongly-typed message parameters
- **Go binding** offers **both** channel-based (idiomatic) AND callback interface patterns
- Go channels for streaming — idiomatic Go concurrency

### Node.js Handling
- **Keep napi-rs** for Node.js — UniFFI doesn't support it, proven solution stays
- **Independent codebases** — js/ uses napi-rs idioms, uniffi/ uses UniFFI patterns, both wrap core/
- **Language-idiomatic** API naming — camelCase for JS, PascalCase for C#, snake_case for Go
- **Unified documentation** — single README with language tabs/sections covering all bindings

### Claude's Discretion
- Whether to reuse existing uniffi/ directory or start fresh (assess current content)
- Sync method timeout handling strategy (per-method vs global config)
- Exact UDL structure and organization

</decisions>
