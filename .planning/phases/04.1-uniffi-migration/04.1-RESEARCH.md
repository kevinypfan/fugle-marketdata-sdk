# Phase 04.1: UniFFI Migration - Research

**Researched:** 2026-01-31
**Domain:** Multi-language FFI bindings, UniFFI architecture, C# and Go code generation
**Confidence:** MEDIUM

## Summary

UniFFI is Mozilla's multi-language bindings generator for Rust, providing a unified approach to FFI through Interface Definition Language (UDL) files. The current uniffi/ crate uses JSON string return values, but phase requirements specify strongly-typed struct models for C# and Go bindings. uniffi-bindgen-cs and uniffi-bindgen-go are third-party generators maintained by NordSecurity, supporting UniFFI 0.28-0.29.

**Key architectural shift:** Moving from csbindgen's C-style FFI (extern "C" with manual callbacks) to UniFFI's IDL-driven approach with automatic type generation. This requires defining all data models in UDL (dictionaries), exposing async functions with tokio runtime attributes, and implementing foreign traits for WebSocket callbacks.

**Primary recommendation:** Define complete UDL with typed records (not JSON strings), use async functions with `#[uniffi::export(async_runtime = "tokio")]`, implement foreign traits for callback-based WebSocket streaming, and create thin C# wrapper for FubonNeo compatibility patterns.

## Standard Stack

The established libraries/tools for UniFFI-based multi-language bindings:

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| uniffi | 0.28.x | Core bindings generator | Mozilla production-proven (Firefox mobile/desktop), defines UDL format |
| uniffi-bindgen-cs | 0.10.0+v0.29.4 | C# code generation | NordSecurity-maintained 3rd party, latest supports uniffi 0.29.4 |
| uniffi-bindgen-go | 0.2.x+v0.28.3 | Go code generation | NordSecurity-maintained 3rd party, supports uniffi 0.28.3 |
| tokio | 1.x | Async runtime | Required for async function support via `async_runtime = "tokio"` attribute |

### Supporting
| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| thiserror | 1.x | Error derive macro | Defining UniFFI error enums with automatic Display impl |
| serde_json | 1.x | JSON serialization | Only if UDL uses JSON strings (not recommended for this phase) |
| async-trait | 0.1.x | Async trait support | Foreign traits with async methods |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| UniFFI | Keep csbindgen for C# | csbindgen: C#-specific, manual FFI, no Go support; UniFFI: unified UDL, multi-language |
| UniFFI | Direct PyO3/napi-rs per language | Per-language: mature, idiomatic; UniFFI: single UDL source but less language-specific optimization |
| Typed records | JSON strings (current uniffi/) | JSON: simpler UDL, runtime parsing; Records: compile-time safety, better IDE support |

**Installation:**
```bash
# Core UniFFI (in Cargo.toml)
uniffi = { version = "0.28", features = ["cli"] }

# Binding generators
cargo install uniffi-bindgen-cs --git https://github.com/NordSecurity/uniffi-bindgen-cs --tag v0.10.0+v0.29.4
cargo install uniffi-bindgen-go --git https://github.com/NordSecurity/uniffi-bindgen-go --tag v0.2.2+v0.28.3

# Generate bindings
uniffi-bindgen-cs uniffi/src/marketdata.udl -o bindings/csharp/
uniffi-bindgen-go uniffi/src/marketdata.udl -o bindings/go/
```

## Architecture Patterns

### Recommended Project Structure
```
uniffi/
├── src/
│   ├── lib.rs                  # uniffi::include_scaffolding!("marketdata")
│   ├── marketdata.udl          # Single UDL defining all types and interfaces
│   ├── rest.rs                 # REST client wrapper (Arc-wrapped)
│   ├── websocket.rs            # WebSocket client with foreign trait callbacks
│   ├── models.rs               # UDL dictionary → Rust struct mappings
│   └── errors.rs               # MarketDataError enum
├── Cargo.toml                  # cdylib + lib, uniffi deps
└── build.rs                    # uniffi build scaffolding
```

### Pattern 1: UDL Dictionary for Typed Models
**What:** Define response models as UDL dictionaries that generate native types in C#/Go
**When to use:** All REST responses and WebSocket message payloads
**Example:**
```
// Source: UniFFI user guide - https://mozilla.github.io/uniffi-rs/0.28/udl/structs.html
dictionary Quote {
    string date;
    string symbol;
    double? open_price;
    double? close_price;
    sequence<PriceLevel> bids;
    sequence<PriceLevel> asks;
    boolean is_open;
};

dictionary PriceLevel {
    double price;
    i64 size;
};
```

**Rust implementation:**
```rust
// Rust struct must match UDL dictionary fields exactly
#[derive(uniffi::Record)]
pub struct Quote {
    pub date: String,
    pub symbol: String,
    pub open_price: Option<f64>,
    pub close_price: Option<f64>,
    pub bids: Vec<PriceLevel>,
    pub asks: Vec<PriceLevel>,
    pub is_open: bool,
}
```

### Pattern 2: Async Functions with Tokio Runtime
**What:** Export async Rust functions that map to Task<T> in C# and (T, error) in Go
**When to use:** All REST API methods requiring network I/O
**Example:**
```
// UDL definition
[Async]
[Throws=MarketDataError]
Quote get_stock_quote(string symbol);

// Rust implementation
#[uniffi::export(async_runtime = "tokio")]
pub async fn get_stock_quote(symbol: String) -> Result<Quote, MarketDataError> {
    // tokio async code here
}
```

**Generated bindings:**
- C#: `Task<Quote> GetStockQuote(string symbol)`
- Go: `func GetStockQuote(symbol string) (*Quote, error)`

### Pattern 3: Foreign Trait for Callback Interfaces
**What:** Define traits implemented by consumer code for event-driven patterns
**When to use:** WebSocket message callbacks, streaming data handlers
**Example:**
```
// UDL definition
[Trait, WithForeign]
interface WebSocketCallback {
    [Async]
    void on_message(StreamMessage message);
    void on_error(MarketDataError error);
    void on_connected();
    void on_disconnected();
};

// Rust accepts foreign implementation
#[uniffi::export]
pub fn new_websocket_client(callback: Arc<dyn WebSocketCallback>) -> Arc<WebSocketClient> {
    // Use callback from C#/Go
}
```

**C# consumer implementation:**
```csharp
class MyCallback : IWebSocketCallback {
    public Task OnMessageAsync(StreamMessage message) {
        // Handle message
    }
}
```

### Pattern 4: Arc-Wrapped Interface Objects
**What:** Object interfaces wrapped in Arc for thread-safe sharing across FFI
**When to use:** Stateful clients (REST, WebSocket) that need interior mutability
**Example:**
```
// UDL definition
interface RestClient {
    constructor(string api_key);
    [Async, Throws=MarketDataError]
    Quote get_quote(string symbol);
};

// Rust implementation stores Arc internally
pub struct RestClient {
    inner: Arc<Mutex<CoreRestClient>>,
}
```

### Anti-Patterns to Avoid
- **JSON String Returns:** Current uniffi/ uses `string quote()` returning JSON. Use typed dictionaries instead for compile-time safety
- **Callback Interfaces (deprecated):** Use `[Trait, WithForeign]` instead of `callback interface` (soft deprecated)
- **Manual Memory Management:** Don't expose raw pointers; let UniFFI handle Arc wrapping
- **Mixed Sync/Async:** Pick one pattern per client type; use async for I/O, sync wrappers if needed
- **Hierarchical Errors:** UniFFI errors must be flat enums; avoid nested error types

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| FFI type marshalling | Manual serde_json + unsafe pointer casts | UniFFI dictionaries + UDL | UniFFI handles serialization, memory safety, null safety automatically |
| Async Task bridging | Manual callback + GCHandle + TaskCompletionSource | `[Async]` attribute + tokio runtime | UniFFI maps Rust Future → C# Task → Go goroutine natively |
| Error code mapping | Integer codes (-1 to -999) with lookup table | `[Error] enum` in UDL | UniFFI generates exception types in C#/Go with message preservation |
| WebSocket callbacks | C function pointers + user_data void* | `[Trait, WithForeign]` interface | Foreign traits provide typed callbacks without unsafe pointer arithmetic |
| Multiple language bindings | Separate csbindgen, PyO3, napi-rs FFI layers | Single UDL file | One UDL generates all bindings; maintain one source of truth |

**Key insight:** UniFFI's entire purpose is automating FFI boilerplate. Manual FFI code (current csbindgen approach) requires:
- Rust: catch_unwind, CString conversion, manual JSON serialization, callback vtables
- C#: unsafe blocks, function pointers, GCHandle lifecycle, JSON parsing
- UniFFI eliminates ALL of this by generating both sides from UDL

## Common Pitfalls

### Pitfall 1: UDL-Rust Struct Mismatch
**What goes wrong:** UDL dictionary field names/types don't match Rust struct → compile error or runtime panic
**Why it happens:** UDL uses camelCase, Rust uses snake_case; serde rename attributes not visible to UniFFI
**How to avoid:**
- Use procedural macros `#[derive(uniffi::Record)]` to generate UDL from Rust structs
- OR manually ensure exact field correspondence between UDL and Rust
- Use `uniffi-bindgen scaffolding` to validate at build time
**Warning signs:** Build errors like "field X not found in dictionary Y"

### Pitfall 2: Async Runtime Attribute Ignored
**What goes wrong:** Async functions don't use tokio runtime, causing executor mismatch errors
**Why it happens:** `#[uniffi::export(async_runtime = "tokio")]` only works on free functions, not trait impls
**How to avoid:**
- Export async functions at module level, not as trait methods
- For trait methods, use `#[async_trait]` and ensure struct has `#[uniffi::export(async_runtime = "tokio")]`
- Test async execution context in integration tests
**Warning signs:** Runtime errors like "no tokio context" or executor panics

### Pitfall 3: Foreign Trait Thread Safety
**What goes wrong:** Foreign trait implementations called from multiple threads → data race or panic
**Why it happens:** UniFFI requires `Send + Sync` but foreign language (C#/Go) doesn't enforce it
**How to avoid:**
- Document thread-safety requirements for callback interfaces
- Use Arc<Mutex<T>> in Rust if callbacks modify shared state
- Test callbacks under concurrent load
**Warning signs:** Intermittent crashes, data corruption in callback handlers

### Pitfall 4: String Size Limits
**What goes wrong:** Large JSON responses exceed i32::MAX (2GB limit) → exception
**Why it happens:** uniffi-bindgen-cs limits string/byte[] length to i32 (2^31)
**How to avoid:**
- Use pagination for large datasets
- Split large responses into multiple smaller dictionaries
- Monitor response sizes in production
**Warning signs:** Exception: "string size exceeds maximum"

### Pitfall 5: Version Compatibility Mismatch
**What goes wrong:** uniffi-bindgen-cs 0.10.0 targets uniffi 0.29.4, but project uses 0.28.x → binding generation fails
**Why it happens:** Third-party generators lag behind core uniffi releases
**How to avoid:**
- Pin uniffi version to match generator: use 0.28.x for current uniffi-bindgen-go
- Check generator tags (vX.Y.Z+vA.B.C format) for uniffi compatibility
- Test binding generation in CI before upgrading
**Warning signs:** UDL parsing errors, missing features in generated code

### Pitfall 6: Callback Interface Deprecation
**What goes wrong:** Using `callback interface` syntax instead of `[Trait, WithForeign]` → works but soft-deprecated
**Why it happens:** Legacy UDL syntax still accepted for backward compatibility
**How to avoid:**
- Use modern `[Trait, WithForeign]` pattern for new code
- Migrate existing `callback interface` to traits during refactor
- Check UniFFI docs for current patterns (https://mozilla.github.io/uniffi-rs/)
**Warning signs:** Deprecation warnings in uniffi-bindgen output

## Code Examples

Verified patterns from official sources:

### REST Client with Async Methods
```rust
// Source: Context from existing uniffi/ + UniFFI docs
// UDL definition
namespace marketdata_uniffi {
    [Throws=MarketDataError]
    RestClient new_rest_client(string api_key);
};

interface RestClient {
    [Async, Throws=MarketDataError]
    Quote get_stock_quote(string symbol);

    [Async, Throws=MarketDataError]
    sequence<Trade> get_stock_trades(string symbol);
};

dictionary Quote {
    string date;
    string symbol;
    double? last_price;
    sequence<PriceLevel> bids;
    sequence<PriceLevel> asks;
};

[Error]
enum MarketDataError {
    "NetworkError",
    "AuthError",
    "InvalidSymbol",
    "ParseError",
};

// Rust implementation
use std::sync::Arc;

pub struct RestClient {
    inner: marketdata_core::RestClient,
}

#[uniffi::export(async_runtime = "tokio")]
pub async fn new_rest_client(api_key: String) -> Result<Arc<RestClient>, MarketDataError> {
    let auth = marketdata_core::Auth::ApiKey(api_key);
    Ok(Arc::new(RestClient {
        inner: marketdata_core::RestClient::new(auth),
    }))
}

#[uniffi::export(async_runtime = "tokio")]
impl RestClient {
    pub async fn get_stock_quote(&self, symbol: String) -> Result<Quote, MarketDataError> {
        let quote = self.inner.stock().intraday().quote()
            .symbol(&symbol)
            .send()
            .await?;

        Ok(quote.into()) // Convert core::Quote → uniffi::Quote
    }
}
```

### WebSocket with Foreign Trait Callbacks
```rust
// Source: UniFFI foreign trait docs + WebSocket pattern
// UDL definition
[Trait, WithForeign]
interface WebSocketListener {
    [Async]
    void on_message(StreamMessage message);
    void on_connected();
    void on_disconnected();
    void on_error(string error);
};

interface WebSocketClient {
    constructor(string api_key, WebSocketListener listener);

    [Async, Throws=MarketDataError]
    void connect();

    [Async, Throws=MarketDataError]
    void subscribe(string channel, string symbol);

    [Async]
    void disconnect();
};

dictionary StreamMessage {
    string event;
    string channel;
    string symbol;
    DataPayload? data;
};

// Rust implementation
use std::sync::Arc;
use tokio::sync::Mutex;

pub struct WebSocketClient {
    inner: Arc<Mutex<marketdata_core::WebSocketClient>>,
    listener: Arc<dyn WebSocketListener>,
}

#[uniffi::export]
impl WebSocketClient {
    #[uniffi::constructor]
    pub fn new(api_key: String, listener: Arc<dyn WebSocketListener>) -> Arc<Self> {
        let ws = marketdata_core::WebSocketClient::new(&api_key);

        Arc::new(Self {
            inner: Arc::new(Mutex::new(ws)),
            listener,
        })
    }

    pub async fn connect(&self) -> Result<(), MarketDataError> {
        let mut ws = self.inner.lock().await;
        ws.connect().await?;
        self.listener.on_connected();
        Ok(())
    }

    pub async fn subscribe(&self, channel: String, symbol: String) -> Result<(), MarketDataError> {
        let ws = self.inner.lock().await;
        ws.subscribe(&channel, &symbol).await?;
        Ok(())
    }
}

// C# consumer usage
class MyListener : IWebSocketListener {
    public async Task OnMessageAsync(StreamMessage message) {
        Console.WriteLine($"Got {message.Event} for {message.Symbol}");
    }
    public void OnConnected() { }
    public void OnDisconnected() { }
    public void OnError(string error) { }
}

var listener = new MyListener();
var client = MarketdataUniffi.WebSocketClient(apiKey, listener);
await client.ConnectAsync();
await client.SubscribeAsync("trades", "2330");
```

### Typed Models vs JSON Strings
```rust
// BEFORE (current uniffi/ - JSON strings)
interface StockIntradayClient {
    [Throws=MarketDataError]
    string quote(string symbol);  // Returns raw JSON
};

// C# consumer parses JSON manually
var quoteJson = client.Quote("2330");
var quote = JsonSerializer.Deserialize<Quote>(quoteJson);
Console.WriteLine(quote.LastPrice);  // Runtime error if structure changed

// AFTER (planned - typed dictionaries)
interface StockIntradayClient {
    [Async, Throws=MarketDataError]
    Quote quote(string symbol);  // Returns typed Quote
};

// C# gets strongly-typed object
var quote = await client.QuoteAsync("2330");
Console.WriteLine(quote.LastPrice);  // Compile-time safety
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| JSON strings in UDL | Typed dictionaries | UniFFI 0.20+ (2021) | Compile-time type safety, better IDE support, no runtime parsing |
| `callback interface` | `[Trait, WithForeign]` | UniFFI 0.24+ (2022) | Arc<T> instead of Box<T>, explicit foreign marker, better docs |
| Manual async scaffolding | `[Async]` + `async_runtime` | UniFFI 0.23+ (2022) | Native Task/Future mapping, no manual executor |
| Per-file UDL | Single UDL per crate | UniFFI 0.21+ (2021) | Build constraint, simpler scaffolding |

**Deprecated/outdated:**
- **callback interface** (soft deprecated 2022): Use `[Trait, WithForeign]` for foreign implementations
- **uniffi_build helpers** (deprecated 0.25+): Use `uniffi::build` crate functions instead
- **Manual scaffolding includes**: Use `uniffi::include_scaffolding!("name")` macro

## Open Questions

Things that couldn't be fully resolved:

1. **C# EventHandler Pattern Compatibility**
   - What we know: Current csbindgen uses `event EventHandler<MessageEventArgs>`, UniFFI uses callback traits
   - What's unclear: Can thin C# wrapper convert `IWebSocketListener` → `EventHandler<T>` efficiently?
   - Recommendation: Test thin wrapper approach; may need dual API (trait callbacks + event wrappers)

2. **Go Channel-Based Streaming**
   - What we know: Context decision says "Go channels for streaming"
   - What's unclear: UniFFI foreign trait → Go channel bridge pattern not documented
   - Recommendation: Implement trait that sends to Go channel; verify goroutine safety in tests

3. **Sync Method Timeout Strategy**
   - What we know: UDL can have both sync and async variants
   - What's unclear: Sync wrapper should block with timeout or infinite wait?
   - Recommendation: Global timeout config (30s default) via client constructor param

4. **Existing uniffi/ Directory Reuse**
   - What we know: Current uniffi/ has JSON string API, needs typed models
   - What's unclear: Migrate in place or create fresh uniffi2/ then replace?
   - Recommendation: Migrate in place; breaking change acceptable per CONTEXT.md

5. **FubonNeo Compatibility Layer Thickness**
   - What we know: Need partial compatibility (method names, async style)
   - What's unclear: How much wrapping needed? Just naming or full API surface adaptation?
   - Recommendation: Start minimal (naming + async wrappers); expand based on integration tests
