# Phase 4.2: Java Binding via UniFFI - Research

**Researched:** 2026-01-31
**Domain:** Java bindings via uniffi-bindgen-java, CompletableFuture async patterns, JNA native library loading
**Confidence:** MEDIUM

## Summary

uniffi-bindgen-java is maintained by IronCoreLabs and generates Java bindings from UniFFI proc-macro interfaces. Unlike the official Kotlin bindings that use kotlinx.coroutines, the Java bindings use standard `CompletableFuture<T>` for async operations, providing idiomatic Java patterns. The generator supports foreign trait implementation, enabling Java classes to implement Rust trait callbacks for WebSocket streaming.

The existing UniFFI infrastructure in Phase 4.1 (proc-macro based, using `#[uniffi::export]` and `#[derive(uniffi::Record)]`) is fully compatible with uniffi-bindgen-java. The Java binding generation follows the same pattern as uniffi-bindgen-cs and uniffi-bindgen-go, using library mode to generate bindings from the compiled cdylib.

**Primary recommendation:** Use uniffi-bindgen-java 0.2.x with the existing uniffi 0.28 proc-macro interface. Generate Java bindings via library mode. Create a thin Java wrapper layer to provide builder patterns, dual sync/async methods, and BlockingQueue-based streaming as an alternative to callback listeners.

### Pattern 1: Builder Pattern for Client Construction
**What:** Static builder class for constructing clients with fluent API
**When to use:** Creating RestClient and WebSocketClient instances (per CONTEXT.md decision)
**Example:**
```java
// Source: Java builder pattern best practices
public final class FugleRestClient {
    private final RestClient inner;

    private FugleRestClient(RestClient inner) {
        this.inner = inner;
    }

    public static Builder builder() {
        return new Builder();
    }

    public StockClient stock() {
        return inner.stock();
    }

    public static final class Builder {
        private String apiKey;
        private String bearerToken;
        private String sdkToken;

        public Builder apiKey(String apiKey) {
            this.apiKey = apiKey;
            return this;
        }

        public Builder bearerToken(String token) {
            this.bearerToken = token;
            return this;
        }

        public Builder sdkToken(String token) {
            this.sdkToken = token;
            return this;
        }

        public FugleRestClient build() throws FugleException {
            if (sdkToken != null) {
                return new FugleRestClient(MarketdataUniffi.newRestClientWithSdkToken(sdkToken));
            } else if (apiKey != null) {
                return new FugleRestClient(MarketdataUniffi.newRestClientWithApiKey(apiKey));
            } else if (bearerToken != null) {
                return new FugleRestClient(MarketdataUniffi.newRestClientWithBearerToken(bearerToken));
            }
            throw new IllegalStateException("Must provide apiKey, bearerToken, or sdkToken");
        }
    }
}
```

### Pattern 2: Dual Sync/Async Methods with CompletableFuture
**What:** Provide both blocking sync and non-blocking async variants
**When to use:** All REST API methods (per CONTEXT.md decision)
**Example:**
```java
// Source: CONTEXT.md decision + CompletableFuture best practices
public final class StockIntradayClientWrapper {
    private final StockIntradayClient inner;

    StockIntradayClientWrapper(StockIntradayClient inner) {
        this.inner = inner;
    }

    // Sync method - blocking
    public Quote getQuote(String symbol) throws FugleException {
        try {
            // UniFFI-generated sync method
            return inner.quoteSync(symbol);
        } catch (MarketDataException e) {
            throw FugleException.from(e);
        }
    }

    // Async method - non-blocking
    public CompletableFuture<Quote> getQuoteAsync(String symbol) {
        // UniFFI-generated async method returns CompletableFuture
        return inner.getQuote(symbol)
            .exceptionally(e -> {
                // Wrap in FugleException hierarchy
                throw new CompletionException(FugleException.from(e));
            });
    }
}
```

### Pattern 4: BlockingQueue for Pull-Based WebSocket Streaming
**What:** Alternative to callback interface using BlockingQueue for thread-safe message consumption
**When to use:** Consumers preferring pull-based over push-based streaming (per CONTEXT.md)
**Example:**
```java
// Source: BlockingQueue producer-consumer pattern
public final class FugleWebSocketClient implements AutoCloseable {
    private final WebSocketClient inner;
    private final BlockingQueue<StreamMessage> messageQueue;
    private final InternalListener listener;

    private FugleWebSocketClient(WebSocketClient inner, int queueCapacity) {
        this.inner = inner;
        this.messageQueue = new LinkedBlockingQueue<>(queueCapacity);
        this.listener = new InternalListener(messageQueue);
    }

    // Pull-based: blocks until message available or timeout
    public StreamMessage poll(long timeout, TimeUnit unit) throws InterruptedException {
        return messageQueue.poll(timeout, unit);
    }

    // Pull-based: blocks until message available
    public StreamMessage take() throws InterruptedException {
        return messageQueue.take();
    }

    // Internal listener that bridges callbacks to queue
    private static class InternalListener implements WebSocketListener {
        private final BlockingQueue<StreamMessage> queue;

        InternalListener(BlockingQueue<StreamMessage> queue) {
            this.queue = queue;
        }

        @Override
        public void onMessage(StreamMessage message) {
            // Non-blocking offer, drops if queue full
            queue.offer(message);
        }

        @Override
        public void onConnected() {
            // Optionally inject connection event
        }

        @Override
        public void onDisconnected() {
            // Optionally inject disconnection event
        }

        @Override
        public void onError(String errorMessage) {
            // Inject error as special message type
        }
    }

    @Override
    public void close() {
        inner.disconnect();
    }
}
```

### Anti-Patterns to Avoid
- **Checked exceptions for runtime errors:** Use RuntimeException hierarchy per modern Java patterns
- **Mutable model classes:** Use final fields per CONTEXT.md decision for thread-safety
- **Direct UniFFI class exposure:** Wrap generated classes in user-friendly API layer
- **Ignoring CompletableFuture exceptions:** Always handle via exceptionally() or handle()
- **Blocking on UI thread:** Document async methods for UI-aware consumers

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| FFI type conversion | Manual JNI/JNA code | UniFFI-generated converters | UniFFI handles null safety, type mapping, memory automatically |
| Async to sync bridging | Thread.sleep loops | CompletableFuture.join() | Standard Java API with proper exception propagation |
| Native library extraction | Custom class loader | JNA built-in extraction | JNA handles OS detection, extraction, cleanup |
| Callback-to-queue bridge | Manual thread management | BlockingQueue + internal listener | Thread-safe, bounded backpressure |
| JSON parsing from data_json | Custom parser | Jackson ObjectMapper | Production-proven, well-documented |

**Key insight:** The Java binding layer should be thin. UniFFI generates correct FFI code; the wrapper provides ergonomic Java patterns (builders, exception hierarchy, Optional<T>) without reimplementing FFI mechanics.

## Common Pitfalls

### Pitfall 1: CompletableFuture Exception Unwrapping
**What goes wrong:** Exceptions wrapped in `CompletionException` confuse error handling
**Why it happens:** CompletableFuture wraps exceptions thrown in callbacks
**How to avoid:**
- Use `getCause()` to access the original exception
- Document that async methods throw CompletionException wrapping FugleException
- Provide utility method: `FugleException.unwrap(Throwable t)`
**Warning signs:** Stack traces showing CompletionException with nested cause

### Pitfall 2: Signed Primitive Interpretation
**What goes wrong:** Java primitives are signed; Rust unsigned types appear negative
**Why it happens:** Java has no unsigned primitive types; i64 for volume can exceed Long.MAX_VALUE
**How to avoid:**
- Use `Long.compareUnsigned()` when comparing volumes/sizes
- Document fields that may contain unsigned values
- Consider using BigInteger for very large unsigned values
**Warning signs:** Negative volumes or sizes in market data

### Pitfall 3: Native Library Loading Failures
**What goes wrong:** `UnsatisfiedLinkError` on startup
**Why it happens:** Native library not found in classpath or system path
**How to avoid:**
- Bundle libraries in correct JNA path: `native/{os-arch}/libname`
- Test on all target platforms (macOS arm64/x86_64, Linux x86_64, Windows x86_64)
- Set `jna.debug_load=true` for debugging
**Warning signs:** UnsatisfiedLinkError mentioning library name

### Pitfall 4: BlockingQueue Overflow
**What goes wrong:** Messages dropped silently when queue full
**Why it happens:** offer() returns false on full queue, message lost
**How to avoid:**
- Use bounded queue with appropriate size (1000+ for high-frequency data)
- Monitor queue size, warn when approaching capacity
- Consider put() for blocking or logging dropped messages
**Warning signs:** Missing messages, gaps in market data sequence

### Pitfall 5: Thread Safety in WebSocket Callbacks
**What goes wrong:** Race conditions in listener implementations
**Why it happens:** Callbacks invoked from background tokio thread
**How to avoid:**
- Document that callbacks are called from non-main thread
- Use thread-safe collections in listeners
- Consider CompletableFuture.runAsync for UI updates
**Warning signs:** ConcurrentModificationException, inconsistent state

### Pitfall 6: Memory Leaks from Unclosed Clients
**What goes wrong:** Native resources not released
**Why it happens:** WebSocketClient not closed, finalizer delayed
**How to avoid:**
- Implement AutoCloseable on wrapper classes
- Document try-with-resources usage
- Consider reference counting for shared clients
**Warning signs:** Memory growth, "too many open files" errors

### REST Client Usage (Complete Example)
```java
// Source: Builder pattern + CompletableFuture best practices
import tw.com.fugle.marketdata.*;
import tw.com.fugle.marketdata.generated.*;
import java.util.concurrent.CompletableFuture;

public class RestExample {
    public static void main(String[] args) {
        // Builder pattern construction
        FugleRestClient client = FugleRestClient.builder()
            .apiKey(System.getenv("FUGLE_API_KEY"))
            .build();

        // Sync usage - blocking
        try {
            Quote quote = client.stock().intraday().getQuote("2330");
            System.out.println("Last price: " + quote.getLastPrice().orElse(0.0));
        } catch (FugleException e) {
            System.err.println("Error: " + e.getMessage());
        }

        // Async usage - non-blocking
        CompletableFuture<Quote> future = client.stock().intraday().getQuoteAsync("2330");
        future
            .thenAccept(q -> System.out.println("Async price: " + q.getLastPrice()))
            .exceptionally(e -> {
                System.err.println("Async error: " + e.getCause().getMessage());
                return null;
            });

        // Wait for async to complete
        future.join();
    }
}
```

### WebSocket BlockingQueue Usage
```java
// Source: BlockingQueue pattern + CONTEXT.md pull-based decision
import tw.com.fugle.marketdata.*;
import java.util.concurrent.TimeUnit;

public class WebSocketQueueExample {
    public static void main(String[] args) throws Exception {
        // Pull-based usage with BlockingQueue
        try (FugleWebSocketClient client = FugleWebSocketClient.builder()
                .apiKey(System.getenv("FUGLE_API_KEY"))
                .queueCapacity(1000)
                .build()) {

            client.connect().join();
            client.subscribe("trades", "2330").join();

            // Pull messages from queue
            while (true) {
                StreamMessage msg = client.poll(5, TimeUnit.SECONDS);
                if (msg != null) {
                    System.out.println("Got: " + msg.getEvent());
                } else {
                    System.out.println("No message in 5 seconds");
                }
            }
        }
    }
}
```

## Open Questions

1. **uniffi-bindgen-java Version Stability**
   - What we know: Currently 0.2.x, marked unstable, IronCoreLabs developing for ironcore-alloy
   - What's unclear: How often breaking changes occur, version pinning strategy
   - Recommendation: Pin to specific commit/tag in CI, test binding generation before upgrades

2. **Record Keyword vs Class for Models**
   - What we know: `generate_immutable_records = true` uses Java records; Java 11 has no records
   - What's unclear: Does uniffi-bindgen-java fallback to final classes for Java 11?
   - Recommendation: Test with Java 11; if records required, document Java 16+ or use manual final classes

3. **Jackson vs Gson for data_json Parsing**
   - What we know: CONTEXT.md marks as Claude's discretion; both work well
   - What's unclear: Whether Jackson's size penalty matters for SDK consumers
   - Recommendation: Use Jackson (Spring Boot default, faster for large payloads); document as optional

4. **Rate Limit Retry-After Header Parsing**
   - What we know: CONTEXT.md specifies `getRetryAfterSeconds()` on rate limit errors
   - What's unclear: How to extract from MarketDataException message
   - Recommendation: Parse from HTTP headers in core if available; fallback to reasonable default (60s)
