# Phase 04.2 Plan 02: Idiomatic Java Wrapper Summary

Java wrapper providing builder pattern, exception hierarchy, and dual sync/async methods for generated UniFFI bindings

## One-liner

Builder pattern wrapper with FugleException hierarchy and dual sync/async methods for CompletableFuture-based API

## Objectives

**Goal:** Create idiomatic Java wrapper that follows Java conventions
**Why:** Generated UniFFI bindings are functional but not idiomatic Java (MarketDataException.RateLimitException, no builder pattern)
**How:** Wrapper classes providing builder, two-level exception hierarchy, dual sync/async methods

## What Was Built

### Exception Hierarchy (Task 1)

Created two-level exception hierarchy matching C# pattern:

```
FugleException (extends RuntimeException)
├── ApiException
│   └── RateLimitException (with getRetryAfterSeconds())
└── AuthException
```

**Key features:**
- Unchecked exceptions (RuntimeException) for modern Java API
- `FugleException.from()` converts generated MarketDataException to appropriate subclass
- `FugleException.unwrap()` extracts and converts CompletableFuture exceptions
- `RateLimitException.getRetryAfterSeconds()` returns nullable Integer (future enhancement)

### FugleRestClient Wrapper (Task 2)

Created main entry point with builder pattern and hierarchical wrappers:

```java
FugleRestClient client = FugleRestClient.builder()
    .apiKey("YOUR_API_KEY")
    .build();

// Sync - blocks and returns result
Quote quote = client.stock().intraday().getQuote("2330");

// Async - returns CompletableFuture
CompletableFuture<Quote> future = client.stock().intraday().getQuoteAsync("2330");
```

**Architecture:**
- `FugleRestClient` (main entry point)
  - `StockClientWrapper`
    - `StockIntradayClientWrapper` (5 sync + 5 async methods)
  - `FutOptClientWrapper`
    - `FutOptIntradayClientWrapper` (3 sync + 3 async methods)

**Builder pattern:**
- `FugleRestClient.builder()`
- `.apiKey(String)` / `.bearerToken(String)` / `.sdkToken(String)`
- `.build()` → creates RestClient via MarketdataUniffi factory methods

**Dual sync/async pattern:**
- Sync methods: `getQuote()`, `getTicker()`, `getTrades()`, etc.
  - Block and return result directly
  - Convert MarketDataException to FugleException via `from()`
- Async methods: `getQuoteAsync()`, `getTickerAsync()`, `getTradesAsync()`, etc.
  - Return CompletableFuture
  - Wrap exceptions via `unwrap()` in exceptionally() handler

### Coverage

**Stock intraday endpoints:**
- Quote (getQuote / getQuoteAsync)

- Trades (getTrades / getTradesAsync)

## Decisions Made

### 1. Unchecked Exceptions (RuntimeException)

**Decision:** All Fugle exceptions extend RuntimeException (unchecked)

**Rationale:**
- Matches C# binding pattern (unchecked exceptions)
- Idiomatic for modern Java APIs (Optional/Stream don't use checked exceptions)
- Checked exceptions force boilerplate try/catch for every API call
- Network/API errors are generally unrecoverable at call site

**Impact:** Callers not forced to catch exceptions, cleaner API usage

### 2. Builder Pattern

**Decision:** `FugleRestClient.builder().apiKey("...").build()` pattern

**Rationale:**
- Idiomatic Java (Retrofit, OkHttp, many modern libraries)
- Flexible - supports 3 auth methods without constructor overloads
- Extensible - easy to add options like timeout, baseUrl in future
- Type-safe - builder enforces authentication required

**Impact:** Clean, flexible API construction matching Java ecosystem conventions

### 3. Dual Sync/Async Methods

**Decision:** Both `getQuote()` (sync) and `getQuoteAsync()` (async) for all endpoints

**Rationale:**
- Matches C# pattern from Phase 04.1
- Clear naming: Async suffix indicates CompletableFuture return
- Flexibility: Scripts/simple use cases prefer sync, production prefers async
- Generated bindings already provide both (quoteSync / getQuote async)

**Alternatives considered:**
- Async-only: Requires .join() for sync, less convenient
- Sync-only: Poor performance for production
- Callback-based: Not idiomatic Java (CompletableFuture is standard)

**Impact:** Dual API pattern covers both simple scripting and production use cases

### 4. Exception Conversion Strategy

**Decision:** `FugleException.from()` for sync, `unwrap()` for async

**Rationale:**
- Sync: Direct MarketDataException → FugleException conversion in catch block
- Async: CompletableFuture wraps in CompletionException, need unwrap()
- Maps flat generated hierarchy to two-level idiomatic hierarchy
- Preserves error type information (Auth, RateLimit, API)

**Implementation:**
```java
// Sync
try {
    return client.quoteSync(symbol);
} catch (MarketDataException e) {
    throw FugleException.from(e);
}

// Async
return client.getQuote(symbol)
    .exceptionally(e -> { throw FugleException.unwrap(e); });
```
