Refactor the WebSocket callback handling to comply with napi-rs 3.x API changes:

1. Change EventCallbacks to store Arc<ThreadsafeFunction<...>> instead of raw ThreadsafeFunction:
```rust
struct EventCallbacks {
    message: Option<Arc<ThreadsafeFunction<String, ErrorStrategy::CalleeHandled>>>,
    connect: Option<Arc<ThreadsafeFunction<String, ErrorStrategy::CalleeHandled>>>,
    // ... etc
}
```

2. Update the `on()` method to wrap callbacks in Arc before storing:
```rust
pub fn on(&self, event: String, callback: ThreadsafeFunction<String, ErrorStrategy::CalleeHandled>) -> napi::Result<()> {
    let arc_callback = Arc::new(callback);
    // ... store arc_callback
}
```

3. Update fire_callback to use Arc::clone() for thread transfer:
```rust
fn fire_callback(callbacks: &Arc<Mutex<EventCallbacks>>, event: &str, data: String) {
    if let Ok(cb) = callbacks.lock() {
        let callback = match event {
            "message" => cb.message.as_ref(),
            // ...
        };
        if let Some(callback) = callback {
            // Clone Arc, not the function
            let callback_clone = Arc::clone(callback);
            callback_clone.call(data, ThreadsafeFunctionCallMode::NonBlocking);
        }
    }
}
```

4. Update import for ErrorStrategy::CalleeHandled (napi-rs 3.x path may differ):
```rust
use napi::threadsafe_function::{ThreadsafeFunction, ThreadsafeFunctionCallMode};
// Note: ErrorStrategy may have moved in 3.x - check and update import path
```

5. Fix any changed enum variants or method signatures based on compilation errors.

Key changes from research:
- ThreadsafeFunction.call() now takes data directly, not Result<data>
- ErrorStrategy may be in different module path
- Use NonBlocking mode to avoid event loop blocking
