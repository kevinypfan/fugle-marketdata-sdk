# v0.3.0 Research Summary: API Compatibility & Configuration

**Project:** fugle-marketdata-sdk
**Milestone:** v0.3.0 - API Compatibility & Configuration
**Researched:** 2026-02-01
**Confidence:** HIGH

## Executive Summary

The v0.3.0 milestone focuses on aligning our SDK constructor APIs with the official Fugle marketdata SDKs (Python and Node.js) while exposing WebSocket configuration options (reconnection, health check) through all language bindings. Research reveals a fundamental gap: our SDK uses **positional string constructors** (`RestClient("api-key")`) while official SDKs use **options object patterns** (`RestClient({ apiKey: 'api-key' })`). This is a **breaking change** that requires careful migration planning.

The architecture analysis confirms that the Rust core already supports all necessary configuration constructors (`with_reconnection_config`, `with_health_check_config`, `with_full_config`). The implementation task is primarily **wiring existing capabilities through FFI boundaries** rather than building new functionality. Each binding layer (Python/PyO3, Node.js/napi-rs, C#/Java/Go/UniFFI) needs to expose options objects that convert to core types at the FFI boundary.

Key risks include: (1) breaking existing user code without adequate migration path, (2) inconsistent configuration defaults across languages, and (3) configuration validation failing at runtime instead of construction time. Mitigation requires deprecation-first approach, single source of truth for defaults in Rust core, and fail-fast validation in constructors.

## Key Findings

### Constructor API Gaps

**Python SDK Differences:**

| Aspect | Official SDK | Our SDK | Priority |
|--------|-------------|---------|----------|
| Constructor | `RestClient(**options)` with kwargs | `RestClient(api_key)` positional | **CRITICAL** |
| Auth methods | `api_key`, `bearer_token`, `sdk_token` kwargs | Positional + static methods | **HIGH** |
| `base_url` config | Supported via kwargs | Not supported | **MEDIUM** |
| `health_check` config | `HealthCheckConfig` class | `ReconnectConfig` (different purpose) | **HIGH** |
| `subscribe()` | `subscribe(dict)` | `subscribe(channel, symbol, odd_lot)` | **CRITICAL** |
| `unsubscribe()` | `unsubscribe(dict)` | `unsubscribe(subscription_id)` | **CRITICAL** |

**Node.js SDK Differences:**

| Feature | Official SDK | Our SDK | Gap |
|---------|-------------|---------|-----|
| Constructor | Options object `{ apiKey }` | Single string | **INCOMPATIBLE** |
| Bearer token | Supported | Not supported | **MISSING** |
| SDK token | Supported | Not supported | **MISSING** |
| Base URL override | Supported | Not supported | **MISSING** |
| Health check config | Supported (WebSocket) | Not supported | **MISSING** |
| Token validation | Exactly one required | No validation | **MISSING** |

### Configuration Options to Expose

**WebSocket Reconnection Config:**
```
max_attempts: u32        (default: 5)
initial_delay_ms: u64    (default: 1000)
max_delay_ms: u64        (default: 60000)
```

**WebSocket Health Check Config:**
```
enabled: bool            (default: true in core, false in official SDKs)
interval_ms: u64         (default: 30000)
max_missed_pongs: u64    (default: 2)
```

**REST Client Config (future):**
```
timeout_read_ms: u64     (default: 30000)
timeout_write_ms: u64    (default: 30000)
```

**Key Decision Required:** Official Python/Node.js SDKs default `health_check.enabled = false`, but our Rust core defaults to `true`. Recommend aligning with official SDKs for API compatibility.

### Implementation Architecture

The architecture follows a clear pattern: **Core owns structs, bindings convert from idiomatic types**.

**Layer 1 - Rust Core (No Changes Needed):**
- `ReconnectionConfig` with builder methods already exists
- `HealthCheckConfig` with builder methods already exists
- `WebSocketClient::with_full_config()` constructor already exists
- All defaults match requirements

**Layer 2 - Python Binding (PyO3):**
- Add `HealthCheckConfig` PyClass
- Modify constructor: `StockWebSocketClient(api_key, reconnect=None, health_check=None)`
- Store configs and use `with_full_config()` in `connect()`

**Layer 3 - Node.js Binding (napi-rs):**
- Add `WebSocketOptions` napi object with nested configs
- Modify constructor: `new StockWebSocketClient(apiKey, options?)`
- Generate strict TypeScript definitions

**Layer 4 - UniFFI Binding (C#/Java/Go):**
- Add `ReconnectConfig` and `HealthCheckConfig` Records
- Add `new_with_config()` constructor
- Test across all three languages

**FFI Boundary Pattern:**
- Use `u64` milliseconds at FFI boundary (not Duration)
- Use `Option<T>` for optional configs
- Convert to core types immediately in binding layer

### Critical Pitfalls

1. **Breaking Existing User Code Without Migration Path**
   - **Risk:** Users upgrade and code breaks with cryptic errors
   - **Prevention:** Deprecation-first approach in v0.2.x, add new constructor alongside old, provide migration scripts
   - **Phase:** Add deprecation warnings before v0.3.0 release

2. **Inconsistent Configuration Defaults Across Languages**
   - **Risk:** Python defaults reconnection enabled, Node.js disabled, C# requires explicit setting
   - **Prevention:** Single source of truth for defaults in Rust core, cross-language configuration tests
   - **Phase:** Define canonical defaults in Phase 1, validate in each binding phase

3. **Configuration Validation Fails at Wrong Time**
   - **Risk:** Invalid config accepted at construction, fails during connection
   - **Prevention:** Fail-fast validation in constructors with helpful error messages
   - **Phase:** Add comprehensive validation in Phase 1 (Core)

4. **Timeout Configuration Confusion**
   - **Risk:** Users set wrong timeout for their use case, production incidents
   - **Prevention:** Self-documenting config names, validation of timeout relationships, documentation guide
   - **Phase:** Address in documentation phase

5. **Reconnection Configuration Creates Infinite Loops**
   - **Risk:** `max_attempts=0` or `initial_delay=0` causes CPU spike, server flooding
   - **Prevention:** Validate minimum values, add jitter to backoff, implement circuit breaker
   - **Phase:** Core validation in Phase 1

## Recommended Approach

### Migration Strategy

**Recommended: Deprecation-First (Non-Breaking Initially)**

1. **v0.2.x (Current):** Add deprecation warnings to string constructors
2. **v0.3.0-alpha:** Add new options-object constructors alongside old
3. **v0.3.0-beta:** Make options-object primary, old constructors deprecated
4. **v0.3.0:** Finalize API, update all documentation
5. **v0.4.0:** Consider removing deprecated constructors

**Alternative: Clean Break**
- Mark v0.3.0 as breaking change
- Provide migration scripts (Python/JavaScript codemods)
- Clear CHANGELOG with "BREAKING CHANGE:" prefix

**Recommendation:** Use deprecation-first for Python/Node.js where user base is larger, clean break acceptable for UniFFI languages (C#/Java/Go) with smaller user base.

### Phased Implementation Order

**Phase 1: Core Config Validation (Estimated: 4-6 hours)**
- Add comprehensive validation to `ReconnectionConfig` and `HealthCheckConfig`
- Align defaults with official SDKs (`health_check.enabled = false`)

- No breaking changes to core constructors

**Phase 2: Python Binding (Estimated: 6-8 hours)**

- Keep string constructor working (deprecated)
- Add cross-language default validation tests

**Phase 3: Node.js Binding (Estimated: 6-8 hours)**

## Requirements Implications

Based on research, v0.3.0 requirements should include:

### Functional Requirements

1. **FR-1:** RestClient and WebSocketClient accept options object constructor pattern matching official SDKs
2. **FR-2:** Support all three authentication methods: `api_key`, `bearer_token`, `sdk_token`
3. **FR-3:** Validate exactly one authentication method provided (error if zero or multiple)
4. **FR-4:** Expose `ReconnectConfig` with `max_attempts`, `initial_delay_ms`, `max_delay_ms`
5. **FR-5:** Expose `HealthCheckConfig` with `enabled`, `interval_ms`, `max_missed_pongs`
6. **FR-6:** Support `base_url` override for both REST and WebSocket clients
7. **FR-7:** Configuration validation happens at construction time, not connection time

### Non-Functional Requirements

1. **NFR-1:** Configuration defaults consistent across all 5 language bindings
2. **NFR-2:** Backward compatibility for string constructor (deprecated) in Python/Node.js
3. **NFR-3:** Clear deprecation warnings with migration guidance
4. **NFR-4:** TypeScript definitions with strict types (no `any`)
5. **NFR-5:** Error messages include field names and valid ranges

### Testing Requirements

1. **TR-1:** Unit tests for all constructor patterns (default, custom, invalid)
2. **TR-2:** Cross-language configuration consistency tests
3. **TR-3:** Integration tests for reconnection behavior with custom config
4. **TR-4:** Integration tests for health check behavior with custom config
5. **TR-5:** Migration script tests (Python/JavaScript codemods)

### Gaps to Address

1. **WebSocket `subscribe()` signature:** Research shows official SDK uses `subscribe(dict)` while we use `subscribe(channel, symbol)`. This is a deeper API change that may warrant separate research or deferral to v0.4.0.

2. **REST client timeout config:** Core hardcodes 30s timeouts. Adding `RestClientConfig` is straightforward but optional for v0.3.0.

3. **Health check default alignment:** Our core defaults `enabled=true`, official SDKs default `enabled=false`. Decision required on which to follow.
